# 💼 Research — Jobs To Be Done

The Jobs to Be Done framework helps understand what job a user is trying to accomplish, beyond the immediate task.

## Core Concept

Users don't buy features—they hire products to do a **job**. A job has three dimensions:

1. **Functional Job**: The practical task to accomplish
   - "I need to store photos securely"
   - "I need to communicate with my team"

2. **Emotional Job**: The feeling they want to achieve
   - "I want to feel organized and in control"
   - "I want to feel connected and included"

3. **Social Job**: How they want to be perceived
   - "I want to be seen as professional"
   - "I want to be seen as creative"

## Output Deliverable

A JTBD Analysis including:
- **User Type**: Who are we designing for?
- **Functional Job**: Practical task to accomplish
- **Emotional Job**: How they want to feel
- **Social Job**: How they want to be perceived
- **Success Criteria**: When is the job "done well"?
- **Current Solutions**: What do they use now?
- **Desired Outcome**: Ideal state after job is completed

Example:

```markdown
# JTBD: Creating a Design Mockup

## User: Product Designer working in fast-paced startup

### Functional Job
"I need to create clickable prototypes quickly so I can test with users before development"

### Emotional Job
"I want to feel confident that my designs will work before investing developer time"

### Social Job
"I want to be seen as efficient and thoughtful in my design process"

### Success Criteria
- ✅ Prototype created in < 30 minutes
- ✅ Can test with real users
- ✅ Can collect feedback quickly
- ✅ Can iterate fast

### Current Solutions
- Figma (60% of time)
- Protopie (20% of time)
- Manual code (20% of time)

### Desired Outcome
- Have validated designs ready for development
- Reduced revision cycles post-launch
- Team confidence in design decisions
```

Use JTBD analysis to identify what users really need, not what we assume they want.
