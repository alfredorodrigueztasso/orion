## PRE-001 (CRITICAL) → v5.3.2 FIXED ✅

### Build Sequencing Gap: build:react skips build:tokens

**ID**: PRE-001
**Severity**: CRITICAL (blocks fresh-clone contributors)
**Effort**: 30 min + 3h preventive fixes
**Release**: **v5.3.2** (COMBINED with #7-9)

### Root Cause

`npm run build:react` calls `turbo run build --filter=@orion-ds/react` directly.
`packages/react/package.json` build script runs `copy:assets` first, which does `cp ../../theme.css assets/theme.css`.
On a fresh clone with no prior `npm run build`, `theme.css` doesn't exist (it's git-ignored, generated by `build:tokens`).

### Failure Chain

1. Developer: `npm run build:react` (fresh clone, no prior setup)
2. `copy:assets` step: `cp ../../theme.css → ../../theme.css` does not exist
3. Without guard, `cp` fails silently with exit code 1
4. Shell `&&` chain stops, next steps don't run
5. Developer sees confusing error with no clarity on what command to run

### Reproduction Steps

```bash
git clone https://github.com/orion-ds/orion /tmp/orion-fresh
cd /tmp/orion-fresh
npm install
npm run build:react    # FAILS with cryptic error
```

### Fix Strategy (Two-Layer)

**Layer 1**: Root `package.json` — Change `build:react` to run `build:tokens` first
```json
"build:react": "npm run build:tokens && turbo run build --filter=@orion-ds/react"
```

**Layer 2**: `packages/react/package.json` — Add node-based preflight guard before `cp`
```json
"copy:assets": "mkdir -p assets/tokens && node -e \"const fs=require('fs');[{p:'../../theme.css',n:'theme.css'},{p:'../../tokens/generated.css',n:'tokens/generated.css'}].forEach(({p,n})=>{if(!fs.existsSync(p)){console.error('ERROR: '+n+' not found. Run npm run build:tokens first.');process.exit(1)}})\" && cp ../../theme.css assets/theme.css && cp ../../tokens/generated.css assets/tokens/generated.css"
```

### Files to Change

- `/package.json` (line 17: build:react)
- `/packages/react/package.json` (line 78: copy:assets)

## PRE-002 (MEDIUM) → v5.4.0

### Preview Modules Path Resolution

**ID**: PRE-002
**Severity**: MEDIUM (blocks docs-site ComponentPreview, not core library)
**Effort**: 2-4 hours (requires architectural decision)
**Release**: v5.4.0
**Status**: Deferred pending Architect decision

### Root Cause

`registry/preview-modules/*.tsx` files are NOT exported from `@orion-ds/react`.
The docs-site tries to import them using relative paths from its context, which doesn't work because the files live in the repo root `registry/`, not in the installed package.

### Three Possible Solutions

- **Option A**: Export preview-modules as `@orion-ds/react/previews` subpath entry
- **Option B**: Build preview-modules into a separate `@orion-ds/preview` package
- **Option C**: Generate static preview snapshots at build time (no runtime import)

## PRE-005 (LOW) → v5.4.0

### JSON Parse Error Handling Missing

**ID**: PRE-005
**Severity**: LOW (developer experience; build fails with cryptic error)
**Effort**: 1 hour
**Release**: v5.4.0

### Root Cause

`scripts/generate-types.ts` and `scripts/generate-types-dynamic.ts` call `JSON.parse()` on token files without try-catch.
If a token file has a syntax error (trailing comma, etc.), the script crashes with a Node.js SyntaxError instead of naming the file and the problem.

### Fix Strategy

Add `readTokenFile()` helper that wraps `fs.readFileSync()` + `JSON.parse()` in try-catch:
- Log the filename and parse error clearly
- Exit with code 1
- Include suggestion to fix the JSON

## PRE-006 (LOW, Reclassified from CRITICAL) → v5.4.0

### require() in Components (Code Quality Refactor)

**ID**: PRE-006
**Severity**: **LOW (FALSE ALARM on production safety)** — Reclassified from CRITICAL
**Effort**: 2 hours (refactoring for clarity, not bug fix)
**Release**: v5.4.0
**Architect Verdict**: ✅ PRODUCTION SAFE. v5.3.1 works correctly in ESM environments.

### Root Cause

Component files (Chart.tsx, Calendar.tsx, CodeEditor.tsx, CollapsibleFolder.tsx, DatePicker.tsx) have synchronous `try { require(...) } catch {}` blocks.

**Key Finding (Architect verified empirically)**:
- ESM `require` is undefined → throws `ReferenceError` at **call-time** (inside try block), NOT at parse-time
- The `catch {}` swallows the error silently
- Module continues loading normally
- The async `checkComponent()` from optionalDeps.mjs is the authoritative availability checker

### Why False Alarm

v5.3.0 regression had unconditional `require()` as primary path without try/catch → crashes at parse-time in ESM
v5.3.1 has `try/catch require()` + async `import()` path → errors are caught, module loads, async path used in production

### Code Quality Recommendation

Refactor component `try/catch require()` blocks to use explicit ESM detection for clarity:
```typescript
let ResponsiveContainer;
if (typeof require !== 'undefined') {
  try {
    ResponsiveContainer = require('recharts').ResponsiveContainer;
  } catch {}
}
```

This makes the intent explicit: "attempt sync load in CJS environments, fail gracefully in ESM."

## PRE-007 (MEDIUM) → v5.3.2 FIXED ✅

### ESM Validation Gate False Assurance

**ID**: PRE-007
**Severity**: MEDIUM (gate ineffective for catching violations)
**Effort**: 1 hour
**Release**: **v5.3.2**
**Status**: ✅ FIXED

### Root Cause

`scripts/validate-esm.sh` (line 72-73) prints `"No require() calls found in ESM modules."` as static text WITHOUT ever searching for require() calls.

The script only runs `node --input-type=module --eval "import('./dist/...')"` for entry points and checks they load.
Because `try/catch require()` doesn't cause load failures, tests pass even when violations exist.

### Fix (v5.3.2)

Add actual grep validation before the echo:
```bash
echo "Checking for require() calls in compiled .mjs files..."
if grep -r "require(" packages/react/dist --include="*.mjs" | grep -v "//"; then
  echo "✗ Found require() calls in ESM modules — ESM spec violation"
  exit 1
fi
echo "✓ No require() calls found in compiled .mjs files"
```

## PRE-009 (MEDIUM) → v5.3.2 FIXED ✅

### ESM Compliance Test Uses Wrong Tool

**ID**: PRE-009
**Severity**: MEDIUM (test reliability/contradiction)
**Effort**: 1 hour
**Release**: **v5.3.2**
**Status**: ✅ FIXED

### Root Cause

`packages/react/src/utils/optionalDeps.test.ts` (lines 158-165) validates "should not use require()" by using `require()` to load the module.

The test is contradictory: it uses the tool it's supposed to prohibit.
Vitest runs in CommonJS, so `require()` works, giving false positive.

### Fix (v5.3.2)

Rewrite test to grep source file instead:
```typescript
it("should not use require() - only import()", async () => {
  const fs = await import("fs");
  const path = await import("path");
  const sourceFile = path.join(__dirname, "./optionalDeps.ts");
  const source = fs.readFileSync(sourceFile, "utf-8");

  const requirePattern = /(?<!\/\/.*)require\(/g;
  const matches = source.match(requirePattern);

  expect(matches).toBeNull();
});
```
